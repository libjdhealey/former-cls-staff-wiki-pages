Macro for Merging Pegasus and OCLC Records

This is a macro for the Connexion Client that inserts a 949 field to tell Pegasus to overlay on the bib record number.

Usage will look like this: 907 .b5738659

Use this for merging local records with fuller cataloging from OCLC.

Created by Library Systems extraordinaire Tom Baker.

'MacroName:949_USE_OVERLAY_LOAD_TABLE

'MacroDescription:Insert 949 *recs=.bov;

'Macro created by Tom Baker

'Macro last updated on Jan 6, 2009

Sub Main

Dim CS As Object

Set CS = CreateObject("Connex.Client")

CS.AddField 1, "949 *recs=bov;"

CS.AddField 2, "907 ."

End Sub

Dec. 6, 2013: Two improvements:

  • make the 907 field say ".b", since we will always be overlaying a bib. record;
  • leave the cursor after the 'b'

'MacroName:OvlyPeg

'MacroDescription:Overlay from Connexion on Pegasus Bib Number

Sub Main

Dim CS As Object

Set CS = CreateObject("Connex.Client")

CS.AddField 1, "949 *recs=bov;"

CS.AddField 2, "907 .b"

CS.CursorColumn = 8

End Sub

Cuttering in Schiller

Basic Cuttering in Schiller

Use C.A. Cutter’s Three-figure [alphabetic]-order table.

Sch: 3 letters + numbers

S and vowels: 2 letters + numbers

Other consonants: 1 letter + numbers

In most cases, cutter by main entry.

Use of OCLC's Dewey Cutter Table Program is ok if limited to 2 numbers for expansion, and call numbers are still shelf-listed locally.

https://www.oclc.org/support/services/dewey/program.en.html

Cuttering in Schiller for Statutes

Sometimes the topical class number is subdivided by “.2” for Statutes, as defined by the Schiller Form Subdivisions Table. It is used only with jurisdiction as the main entry.

In this case if a country is the main entry, cutter by the first corporate added entry (often the publisher.)

If there are no corporate added entries, cutter by A2.

If A2 is already assigned in the shelf-list, cutter by A3. If A3 is already assigned, cutter by A4, and so on.

If you get up to A6, take precautions to leave enough space in the shelf-list. Start assigning A62, A63, etc.

It may be helpful to look at the same class number in a bigger country for examples of Codes, such as France or Germany.

New books list procedure

Below describes how the monthly recent acquisitions web page is generated. The procedure is carried out at the very beginning of each month.

For the Recent Acquisitions RSS feed, see Recent Acquisitions - New Books RSS Feed.

The newbooks list requires one file of input and produces two files of output.

Input File:

  • An exported review file with recent cataloging

Output Files:

  • A javascript file (e.g. march09.js) in a specified format
  • An HTML file serving as a template to display the javascript file, e.g. march.2009.html. This will be viewable at http://library.law.columbia.edu/newbooks/march09.html

The procedure makes the following assumptions:

  • The new books list uses the /xrecord Command Link; see XRECORD COMMAND LINK CS Direct for more details. The /xrecord Command Link requires that the XRECTYPE WWW-option be active for bib records in the opac. This is set in the 'wwwoptions' files. As of Dec. 6, 2010, XRECTYPE in the 'live' webpac is inactive, if activated, the setting would be 'biec'; XRECTYPE in the 'staging' webpac is live and is set to 'bicae'. b=bibliographic, i=item, c=checkin(holding), a=authority, e=resource record types; only these types can be delivered in xml format.
  • To carry out this procedure you need:
    • access to Millennium Create Lists;
    • access to libdev.law.columbia.edu via PuTTY and via a Samba connection to \\libdev; instructions related to this item are at Tech services scripts
    • a Samba connection to \\www2\libr-web. (This location can be mapped to your desktop machine; usually mapped to I:\);
    • editing access (as a member of the Library group) to the main law-school website

The Python program transforms the Millennium file into the javascript format and excludes titles that don't meet the new books criteria.

Contents
  1. In Pegasus (Millennium, Create Lists)
  2. In the 'newbooks' folder (Samba connection to \\libdev)
  3. In libdev.law.columbia.edu (PuTTY connection)
  4. Using Samba connection to \\libdev AND Samba connection to library.law.columbia.edu
  5. In the law school website
In Pegasus (Millennium, Create Lists)
  • Create a review file of last month's newly received and cataloged titles. E.g. for April 2009 list, search for Order records with a "Cdate" greater than or equal to 03-01-09. To be sure you don't find records from the beginning of the next month, your date range would be greater-than-or-equal-to the first of the month AND less-than-or-equal-to the last day of the month.
  • Export this list to your workstation. Export only the bib number for each of these records. Retain quotes as 'text qualifiers'. (The other defaults are OK, too.)
In the 'newbooks' folder (Samba connection to \\libdev)
  • Using the Samba connection to \\libdev, copy (click&hold on the icon, drag) your file of exported bib. record numbers to the 'data' folder. You will find the 'data' folder inside the 'newbooks' folder.
  • In the 'newbooks' folder, edit the config.ini file. config.ini will look like this:

[New books]

month: October <- Adjust.

year: 2009 <- Adjust as needed.

cutoff year: 2007 <- Adjust as needed.

Open With: WordPad or Notepad++.

  • Adjust the values to reflect the month and year you are working on. The cutoff date is the last date of publication we want to appear in the newbooks list. With a cutoff year of 2007, no books with a publication date from 2006 or before will appear in the list.
In libdev.law.columbia.edu (PuTTY connection)
  • Use PuTTY to make your SSH connection to libdev.law.columbia.edu.
  • Run the the new books program:

newbooks ./data/exported_new_titles.txt

Of course, be sure you substitute the real file name for "exported_new_titles.txt". The output will be placed in libdev's newbooks\web\ directory.

Using Samba connection to \\libdev AND Samba connection to library.law.columbia.edu
  • Copy (click+hold, drag) the outputted html and js files to the library.law server. They should go in the 'newbooks' directory of library.law.columbia.edu. As stated above, the most convenient access is via Samba connection to \\www2\libr-web. This requires permission on the server.
  • Test in your browser to make sure the file is viewable. Make sure both files have the group read permission.

OCLC holding codes

A description of our library's holding codes in the OCLC database.

This came up when we were setting put the YBP Shelf Ready process.

Copy of current Diamond Law Library holding codes

From an OCLC email to Tom Baker on 11/2/2009:

A holding library code is a four-character code that identifies one location or collection within your institution. Each holding library code defines a collection or category of material. Holding library codes, in combination with your profile, determine catalog card production characteristics. They can also be used to manipulate your database for other purposes. For example, if a collection in your library were moved to another library, OCLC would be able to use the holding library code for that collection to locate each item in the collection and flip its OCLC symbol to the new library's symbol.

The holding library code is stored in the MARC record in the 049 field. Field 049 is not part of the standard MARC 21 format. Field 049 is not retained in the master record. It is retained in archive records, institution records, OCLC-MARC records, and exported records.

When you retrieve a record, the system displays your default holding library code in subfield �a. If the default code is XXXM, the system displays:

049 XXXM

If you want to change the holding library code, you may overtype the default code in uppercase or lowercase. When you send Produce commands, the system uses your profile to create the correct cards for holding library codes in field 049.

Defining holding libraries and codes:

Each collection in your library that has unique processing requirements (such as a separate shelflist or a unique location stamp) is a holding library. During profiling, you define your holding collections and assign each a holding library code. For example, a general collection, a music collection, and a science collection each has a holding library code.

Holding library code specifications:

For each holding library code, your profile defines the following:

The types of cards produced (main entry, title entry, subject entry, shelflist, and so forth)

Location stamps, notes, and tracings to appear on each type of card.

The receiving catalogs (for example, public catalog, main shelflist) into which each card is filed. Cards for several holding libraries may file into one receiving catalog. For example, cards for a music collection may file into the public catalog along with cards for the general collection and the science collection. In addition, the music and the science collections may have cards that file into separate department catalogs.

The sort order (the sequence in which the cards are delivered) for each receiving catalog that receives cards from the holding library code. For example, alphabetic order, call number order (shelflist only), user-data order, LCCN order or Input stamp/call number order.

The classification scheme.

The subject headings authority.

Added copy cards. Each holding library code may be profiled to produce added copy cards. How you profile your holding libraries for added copies depends on your workflow, how you process multiple copies, and so forth.

Extra copy cards. All holding library codes are profiled for xc cards that allow production of an extra copy card for a specific holding library code when needed.

zc cards. All holding library codes are profiled for zc cards, which are produced when an error occurs in card production.

For more information about proper construction of the 049 field, navigate to: http://www.oclc.org/us/en/bibformats/en/0xx/049.shtm

Description

OCLC offers the Bib. Record Notification service as part of its cataloging package.

The service notifies a member library if one or both of two changes happen to a WorldCat master record to which the library's holdings symbol is attached:

  • If there is a change to the record's Encoding Level (ELvl). This is a one-byte fixed field contained in the Marc Leader.
  • If there is a change to the 505 field. This can be the addition of a 505, the upgrading of a 505, or the deletion of a 505 field.

Notification consists of three parts: a Marc file of changed records, a report file listing the changes and showing why a particular record was included, and a statistical file, which contains the count of records changed.

Notification is received the day after the changes are made. Thus we can expect to receive notifications virtually every day of the year.

We started receiving these notifications beginning from Friday, Nov. 5, 2010.

Location of files in OCLC

http://psw.oclc.org/ , under Bibliographic Record Notification records and reports.

  • Choose Download records and reports, authenticate (100-056-217//pegasus), then choose the line OCLC Bibliographic Record Notification records and reports. The first group of files are the records themselves, in Marc format, one file for each day. The second group of files are the reports on each file, in 'txt' format. The third group are statistical reports.
Location of files locally

G:\library\TECHSERV\BIBCON\OCLC BibNotif MARC files\
G:\library\TECHSERV\BIBCON\OCLC BibNotif Reports\

Download the files, and move them to the proper directory. You may wish to 'break' the Marc files (i.e. make them readable) using MarcEdit. Hint: before you try to 'break' the Marc files, rename them by changing the extension '.bin' to '.mrc'.

Our use of the files (initial period, Nov. 2010-Feb. 2011- )

Currently we are using only the Reports. The Reports directory contains a Processed directory into which the reports are put once they have been processed.

Tom has gathered past files into directories labelled by time periods, in order to keep the directory listing neat.

Our Profile

OCLC supplies records according to a profile which the library fills out, and can change at any time, in accordance with the library's processing requirements.

Note that at present (Nov. 2010- ) the library's profile choices affect only notifications regarding changes in encoding level. As we have asked to receive notifications of changes to the 505, we will receive all such changes, including changes to a 505 which we ourselves have made.

MarcEdit and OCLC Bib. Notification Using MarcEdit to make a checklist of incoming bib. records with 019 fields

(The presence of 019 indicates that OCLC has merged duplicate records in its database. The 019 contains the OCLC numbers of the records that have been merged into the single remaining record.)

(This procedure demonstrates two very useful MarcEdit procedures: (1) selecting and extracting certain Marc records from a file of Marc records, and (2) selecting and exporting a subset of data from a file of Marc records, using the comma-separated-values (.csv) file format. (.csv files can be imported into Excel. See Export a list from Pegasus, in this wiki, for instructions. Or, right-click on your .csv file, and choose Open With...)

See MarcEdit for basic information about MarcEdit.

  • Determine which files contain records with 019.
    • Use MarcEditor to open the mnemonic Marc record (.mrk).
    • Go to Reports > Field Count. You will see right away if your file contains any records with 019. There is no need to print anything!
  • Extract these records from the file.
    • Start from the MarcEdit main screen.
    • Tools > Select MARC Records > Extract Selected Records.
    • Source MARC File (entry box, at the bottom): click on the folder icon and locate your .mrc file.
    • Change display field to 019$a. Then click on Import File (upper right-hand corner).
    • Select those records that contain data in 019. Click on Export Selected.
    • The file window will appear. Create your file name. Model: D101105.019.mrk. Note that you are creating a mnemonic ('broken', .mrk) file.
    • (NB: It looks as if you could choose to create a .mrc file here, but it isn't working properly. [Nov. 2010])
    • Open your .mrk file and compile it into .mrc. The next step expects input of a .mrc file. You are now done with this phase. Close this MarcEdit Window.
  • Prepare checklist from the file you have just created.
    • Start from the MarcEdit main screen.
    • Tools > Export Tab Delimited Records
    • In the upper box, find the .mrc file you created in the previous step.
    • In the lower box, choose the location and filename (.csv) for the file you are creating.
    • Retain comma as your delimiter. Click on Next for next screen.
    • On the second screen, choose the field and subfield you want to export from each record. For a checklist of OCLC numbers where 019 is involved, these seem good:
      • 001
      • 019
      • 245$a (You can just type your number into the box. Or select from the dropdown list.) For each, click on Add field.
    • Click on Export. Your file will be created.

On CHAR ENC field (Leader Byte 9)

revised 8-27-2015

This section discusses the character encoding of our records, and the CHAR ENC field in these records.

Our internal database storage is Unicode (UTF-8). We can accept records in either Marc8 or Unicode (UTF-8) encoding. MARC records in Unicode are to have the Leader Byte 9, CHAR ENC, set to 'a', and MARC records in Marc8 encoding are to have Leader Byte 9 set to blank " ".

We have (and should have) @leader_utf8 set to "y", and @diac set to "" (null). We do not need @diac_sub_table; that is for databases that do not use Unicode (UTF-8) storage. TB's commenting that line out in our active load tables, starting today with 'b' and 'bov'. 'pcat' was already correct, and is instructive:

We had all three lines active there, and diacritics were not being handled correctly. Transmission is in Marc8, and with @diac_sub_table="utf8" active, the Marc8 diacritics were getting mangled. With only @diac="" and @leader_utf8="y", incoming Marc8 diacritics are being handled correctly.

Nevertheless, it seems best to set Connexion to bring about Export (download) in UTF-8 encoding. Please see OCLC_Connexion_Settings#Setting_up_Export_in_UTF-8_Encoding

Our load tables will henceforth be set this way: 

|||0|0| | |0|n|G|0|@diac=""
#|||0|0| | |0|n|G|0|@diac_sub_table="utf8"
|||0|0| | |0|n|G|0|@leader_utf8="y"

See Load Profile Training Manual (2012 ed.), @diac and @diac_sub_table (both on p. 40), and @leader_utf8 (p. 45).

See also this post to load-profilers@listserv.iii.com, from David Jones, of Santa Clara University: 

On Thu, Jun 6, 2013 at 2:40 PM, Ann Kebabian <akebabian@colgate.edu> wrote:
> convert Marc8 to Unicode when loaded?  Should I rewrite the load tables that
> are expecting Marc8 to expect UFT8?

We're not on Sierra, but when we did the Unicode conversion on our
Millennium system, we had to go into all of our load profiles and
remove any "@diac_sub_table" lines, add the "@leader_utf8" and set it
to 'yes', and set the "@diac" equal to blank: [TB: he means "null" and his example uses null.]

|||0|0| | |0|n|G|0|@diac=""
#|||0|0| | |0|n|G|0|@diac_sub_table="utf8"
|||0|0| | |0|n|G|0|@leader_utf8="y"

Non-ASCII code points in records without the leader position 9 set to
"a" are assumed to be in MARC8 are are translated by the table into
Unicode code points and then encoded with the curly braced form. Those
non-ASCII characters in records with the leader position 9 set to "a"
are just encoded directly with the curly braced form.

Note that not all CHAR ENC fields show 'a' for Unicode. Instead they are a mix of ' ' for Marc-8 and 'a' for Unicode. This happens because the field tells a receiving system what encoding system the record is using in transit. It doesn't give the receiving system an instruction as to what encoding system to use to store the record.

Unicode in our bib. records
  • As TB remembers it, we did brief experiments in transmission of records using Unicode encoding, and problems were found with some Japanese characters and/or symbols. This is why we stuck with Marc-8 for transmission of records from OCLC until 2013. See below.
  • Our original interest in Unicode came about because we had to reconcile our holdings with OCLC ca. 2006. In preparing for that procedure, we changed our internal database storage. Up until that point it had been a combination of Marc-8 for Latin-letter records with CCCII for CJK records. It seemed advisable to convert our storage to Unicode, for reasons explained below.
Holdings discrepancy
  • At the time that RLIN was taken over by OCLC, we discovered we had a problem: our CJK holdings in OCLC were very much smaller than what was reported in RLIN. (We didn't realize that this was happening because we were uploading our records only to RLIN; RLIN passed them along to OCLC.) This discrepancy in holdings made it necessary to reconcile our system's records of our holdings with OCLC.
  • The problem was the way our CJK records had been encoded. They had been encoded using a system called CCCII. RLG accepted CCCII, but OCLC did not. A discrepancy in holdings had therefore developed over time, in that OCLC simply wasn't recording our CJK holdings.
OCLC Reconciliation
  • In the OCLC Reconciliation procedure, the library uploads all its current records to OCLC. If OCLC is missing a record, it will have it now. If OCLC already has the record, the date associated with the holdings symbol will be updated. If OCLC had a record, but the library no longer had the item, the holdings symbol will NOT be updated. All the holdings that are not updated are then deleted, because the library no longer has the item. OCLC and the library now agree as to what the library actually holds at present.
  • In order to reconcile our holdings we needed to upload all of them to OCLC. For OCLC to accept them all, we needed to upload Unicode records. So we changed our database storage. For the records to be recognized as Unicode, the CHAR ENC fields had to be 'a'. So we changed all these fields, but we do not change them in ordinary processing.
CHAR ENC in our records as of Oct. 10, 2013
  • On Oct. 10, 2013, mid-day, our system contained 500,613 bibliographic records. Of these, 463,579 had CHAR ENC fields of 'a' (unicode), and 36,691 of ' ' indicating transmission from OCLC in Marc-8. The remainder, 343, were non-Marc bibs.
  • It may be time to revisit the issue of transmission from OCLC to us, to see if Unicode transmission gives satisfactory results now.
    • (Note that Sierra actually uses the term 'UCS/Unicode' as the name of the Unicode system in offering the choice of encoding systems for the CHAR ENC field. 'UCS' means Universal Character Set, defined by International Standard ISO/IEC 10646. Since 1991, the Unicode Consortium has worked with ISO to develop the 'Unicode Standard' ("Unicode") and ISO/IEC 10646 in tandem, so they're the same thing in practical terms.)
Backstage loading: problems we encountered (June 6, 9- , 2014)

1. Overlay was being rejected for records with LOCATION=multi where a cited location occurred on more than one line (occurred more than once) in the "Number of copies" dialog box. 

The solution was to reduce these multiple occurrances of a single location to one line, with the
"number of copies" being the sum of the number of copies on the previous 'multiple' lines.
Note that this may create some confusion later on: if the previous display was indicating that
c1 was in ofr, c2 was on 3rd, and c3 was in ofr again, the Locations field may now imply that
c1 and c2 are at ofr and c3 is on 3rd. Some clean-up may be necessary.

2. There is a loading-procedure Global Variable command called "ins" which allowed the operator of an old RLIN terminal to enter his or her initials (password) where it might be needed to load records. The documentation cites the example of assigning incoming records to accounting units. In the context of automated record loading, this command was operated by menas of the programmatic insertion (@password) of a 910 field, whose |a was to contain the initials to be used. '910' was causing a problem of us because certain of our records have '910' fields. 

The solution was to change the Global Variable command so that the password would be inserted into 923 |a.
We do not use the 923 field. It may also be possible, and advisable for us, to comment out the '923' line
(and the '910' line of other load tables), since as far as I know we do not use this type of passwording
in loading records.
